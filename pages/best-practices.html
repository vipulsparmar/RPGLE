Best Practices

Writing modern RPGLE isn't just about using free format — it's about adopting professional software engineering standards. Follow these principles to write clean, maintainable, and efficient code.

1. Embrace Full Free Format

Adopt **free on the first line of every source member. Avoid mixing fixed and free format whenever possible. It's the standard for IBM i modernization.

✅ No column limits

With **free, you can use any column from 1 to 200+ (depending on your editor settings). Don't restrict yourself to columns 7-80 anymore!

2. Use Meaningful Naming

RPGLE supports names up to 4096 characters. Stop using 6-character legacy names (e.g., CUSTNM). Use camelCase or PascalCase for readability — just like Python or Java.

Legacy (Bad)
dcl-s CSTADR char(50);
dcl-s WBAL   packed(11:2);
Modern (Good)
dcl-s customerAddress char(50);
dcl-s workingBalance  packed(11:2);

3. Modular Design

  • Avoid Global Variables: Pass data into procedures via parameters.
  • Encapsulate Logic: One procedure should do exactly one thing.
  • Use Service Programs: Group related procedures into service programs (*SRVPGM) for reuse across multiple applications.

4. Performance Tips

  • Varchar for Web/SQL: Use varchar instead of char when building strings or working with web services to avoid constant %trim() calls.
  • BIFs vs OpCodes: Prefer BIFs (like %scan, %subst) over legacy operation codes.
  • Index correctly: Ensure your files have logical views or SQL indexes that match your setll and reade operations.

5. Modern Error Handling

Use monitor blocks for structured error handling instead of legacy "(E)" indicators or *PSSR.

RPGLE
monitor;
  result = value1 / value2;
on-error 00102; // Divide by zero
  result = 0;
on-error;
  logError('Unexpected error');
endmon;

6. Self-Documenting Code

Use // for comments. Add a header block to your procedures explaining the purpose, parameters, and return values.

📝 Pro Tip

Write your code so that the logic is clear from the variable and procedure names. Comments should explain why you're doing something, not what you're doing (the code already tells us what).
